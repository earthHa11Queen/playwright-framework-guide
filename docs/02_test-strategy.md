# 514.テストについて

## 共通する条件や内容について

- テストの区切りとしては以下である。
	- 単体画面テスト
	- 単体APIテスト
	- 結合画面テスト
	- 結合APIテスト
	- 総合テスト
	- 総合性能負荷テスト
	- 総合ペネトレーションテスト
- テストで使用・生成するものとしては以下である。
	- テスト仕様書
	- テストケース(以下ケースと呼称する)
	- テストデータ
	- テストコード
	- テストツール
	- 生成用プロンプト-鏡の原理全容
	- 生成用プロンプト- Mirror Frame Work 
	- 生成用プロンプト- Mirror Frame Work E2E
	- 生成用プロンプト-隣接テーブルとジョインテーブル
	- 生成用詳細設計書
	- 生成用ソースコード
	- 生成用テストコード設計書
	- 生成用各種出力用ファイルテンプレート
	- DOM解析および解析データによるコードやデータの作成ツール
- テストで使用・生成するものの関係性や内容説明
	- テスト仕様書は複数のテストケースをまとめたファイルであり、テーブル形式である
	- テストケースはケースナンバーで識別される、テスト仕様書のレコードである
	- テストデータとテストコードは分離して管理生成される
	- テストコードは
		- テストスクリプト
		- テスト用汎用メソッド
		- テスト用共通メソッド
		- テスト用固有メソッド
		- テストデータメソッド
		- 以上5種類のテストソースコードまとめた呼称である
	- テストスクリプトは、その他のテストメソッド群を順番に呼び出すことで、テストケースの操作を自動化する
	- テストスクリプトファイルはテスト仕様書ファイルと1対1の関係を持つ
	- テストスクリプト内にはテストケースが全て内包されるべきである
		- つまり、テスト仕様書としては膨大な数のテストケース数を持つようにテスト仕様書ファイルを作るべきではない。
		- もっと言えば、テスト仕様書のケース数が膨大な状態とは、ケースの観点をごちゃごちゃと混ぜ込み過ぎている状態であり、すなわち全ての観点を見ている気になっているテスト仕様書である。
		- テスト仕様書は、ある単一の観点を実現するために、結果的に他の観点も取り入れた方が効率良かった、場合にのみ他の観点を盛り込むべきである。
		- つまり本来は必要な観点を整理した上で、ごちゃごちゃと混ぜるのではなく、意図しつつ結果的に観点が網羅されたの点検が必要なものであるから、ごちゃごちゃ混ざってますなんてことはあり得ない(職務放棄)。
	- テスト用汎用メソッド
		- 何かの特定のWebアプリに依存しない、どんなWebアプリのテストでも使う可能性があるようなメソッドのこと
	- テスト用共通メソッド
		- 特定のWebアプリに依存するが、テスト各工程には依存しない
	- テスト用固有メソッド
		- 特定のWebアプリ専用のメソッド群であり、テスト各工程で最も頻繁に呼び出されるテストメソッドの実体である
	- テストデータメソッド
		- 特定のWebアプリ専用のメソッド群であり、テスト各工程毎に共通でありつつ、最適化する必要もある。
- テスト仕様書は次の要素で構成される。
	- ケースナンバー
	- 大分類
	- 中分類
	- 小分類
	- 観点名ないし観点ID
	- ケース内容
	- 操作手順
	- 確認内容
	- 確認手順
	- 想定結果
	- 想定結果判断基準・判断手順
	- テスト実施記録(個人開発ならば不要としても可)
		- 予定日
		- 実施日
		- 実施者
		- 実施結果
	- テスト再実施(個人開発ならば不要としても可)
		- 再実施要否
		- 再実施判断日
		- 再実施判断担当者
		- 再実施テスト仕様書ファイル名ないしファイルパス
	- 備考
- ケースでは以下を作成・作業する
	- パターンの洗いだし
	- 使用する観点のピックアップ
	- パターンの言語化=文章化=ケース化
	- ケースナンバーやケース分類の定義
	- 単位毎のケースファイル作成
- テストデータ(以下データと呼称する)の対象となるデータは以下
	- テーブルデータ
	- モックデータ
	- 正常系投入・入力データ
	- 異常系投入・入力データ
	- であり、モックはJSONで、それ以外はCSVで活用する
- テストコード(以下コードと呼称する)
	- コードは独自に決めたフォーマットで記載していく
	- 画面やAPI開発時のコーディング規約にしたがった命名規則やコーディングをテストコードでも行う
- 前提条件としては以下である。
	- AmazonQ, Amp, ChatGPT, Copilot, ChatGPT-APIなどのプロジェクトで使用できるものに限ったAI製品として活用する。
	- 基本設計や詳細設計、APIや画面のコーディングはある程度終わっているものとし、テストによってバグや実装ミスを弾き出す段階に至っているないし至る手前を想定する
	- バックエンドの言語はPython, Java Spring Boot, PHP, Goである
	- フロントエンドの言語フレームワークは、React, Angular , vueの3種である。
	- アーキテクチャはクリーンアーキテクチャを採用している
	- Docker環境上に構築している
	- データベースはPostgreSQLかMySQLである
	- 自動テストライブラリの活用を前提としている
次に結合テストについて
- 共通部分
	- どちらでもPlaywright を使用するがAPI関しては一部単体テストで用いるpytest や~Unitなどのテスト自動化ライブラリを使用する。特にPlaywright では不可能な箇所に関しては無理にPlaywright で実施しようとはせずに実現しやすくかつテストとして意味が見いだせるものを迷わずに使用する
	- モックは適宜使用するが、それは入出力で必要な時などに限定して使用する
- 結合画面テスト
	- 主に画面遷移や業務ロジックなどをテストする
	- 基本的にはPlaywright のみをテストツールとして使用するが、UWSCなどRPAが必要な場合は適宜併用する。その際はPlaywright から呼び出す形を取る。
- 結合APIテスト
	- 一つの機能に関するAPI間を網羅的にテストする、これはJUnit などで行う
	- 画面からの疎通テストとしてかつ機能の結合および連結としてPlaywright を使用する
	- パターンとしては次のようなものがある。
    	1. 画面からリクエスト/レスポンスを確認
    	2. 画面からAPIを叩いた結果、ログファイルが出力されるか？
    	3. 画面からAPIを叩いた結果、ウイルスチェックが走っているか？
    	4. 画面からAPIを叩いた結果、ログが書き込まれ、内容は設計通りか？
    	5. 画面からAPIを叩いた結果、バッチが走っているか？
    	6. APIをCurlコマンドで叩いた結果を上記のパターンと同じく実施
    	7. その他認証トークンなどがあれば実施
    	8. 上記の内容の正常系と異常系全組み合わせ。特に異常系の場合は設計通りに応答ないし挙動するか？
次に単体テストについて
- 単体エンドとはモジュールテストを指し、基本的には単体APIと称する
- 単体フロントは画面単体テストを指し、基本的には単体画面と称する
- 単体テストにおいては原則C1網羅であり（C1カバレッジ率100%）、カバレッジ率100%を下回る場合というのは、テストコードの実装が非常に困難な場合かつ、実装する場合時間やトークン消費が割に合わない場合に限って許可される。
- ただし、カバレッジ率は最低でも90%以上とし、原則99%に限りなく近づけるものとする。つまりやむを得ない場合に限ってカバレッジ率100%を下回ることが許容される。
- 単体画面
	- Playwright で実行する
	- 画面の遷移はテスト対象外である
	- ボタンやテキストボックスなどの部品が機能するかのテストをベースに、入力文字数制限が反映されているかのチェック、禁則文字のチェックなど各部品と画面単体における異常系を網羅する。
- 単体API
	- Pytest, JUnit, PHPUnit などバックエンド言語に沿うような自動テストライブラリをしようする。
	- Playwright は使用しない
	- モジュールが機能するかを基本的にモックを用いてテストする
総合テストについて
- 分類は以下の通りである。
	- ブラウザバリエーションテスト:通例的にブラバリテストと呼ぶ。ブラバリテストでは、PCで有名な４種類のブラウザ、GoogleChrome・MicrosoftEDGE・サファリ、FirefoxをWindowsOSとMacOSのPCで実施する。また、スマホ対応のWebアプリやネイティブアプリとも連携するシステムである場合は、AndroidスマホとiPhoneでGoogleChromeとサファリでそれぞれ行う。これはPlaywright で可能と考える。ただし、Playwright だけでは困難であるため、UWSCで代表されるようなマクロでのアプリケーション操作自動化ソフトウェアとも連携を取りテストする必要がある。特にブラウザの拡張機能と連携したWebアプリであればサファリでのテストは困難を極める。そのためPlaywright でコーディングしたテストをAppleScript にAIを使ってコンバートするといった一手間をかけることでコーディングやテストを容易にする。
	- 業務シナリオテスト:業務シナリオに沿った画面遷移を行うテストである。単体画面テストと連結画面テストの延長線上にあるような機能テストと説明しても正しいとは思う。これもPlaywright で可能と考える。
	- 性能テスト: 主に負荷テストとなるような、大量インサートや短時間大量リクエスト、それら複合テストにより要求に沿う処理が時間内に終わるかどうか？を検証する。Playwright の並列処理により可能と考えるが、このテスト用に端末やサーバを用意するのがベストと考える。ただ可能な限りJMetter などの製品を使用する。
	- セキュリティテスト:主にはインジェクションテストがメインとなると考える。つまりはPlaywright で悪意のあるコードをテストデータとして用意しておき、普通に画面から自動入力することで十分可能と考える。
	- ペネトレーションテストは外注すべきである。
最後に
- 以上のことをAIの生成能力で置き換えつつ、ドキュメントの作成フローなどはプロセス自体を変革することを考えている。例えばパターンの洗いだしなどはテスト技法をAI伝えて技法に沿った洗いだしを行わせる。
- また、テストの目的はホワイトボックスとブラックボックスの両方を迅速に作成することである。そのためにはインプットデータとして詳細設計書のみやマーメイドで記載したユースケース図のみなど、実装を見せないままケースを作成させるなどの工夫を行う。
